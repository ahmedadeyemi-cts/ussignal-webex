US Signal Webex Customer Portal — Executive Guide

This portal is a single pane of glass that consolidates operational data from Cisco Webex Control Hub and related Webex platform sources into customer-ready dashboards. It is designed so customers can quickly understand posture, prioritize actions, and engage US Signal with shared context to accelerate troubleshooting, audits, and change execution. Built-in automation and intelligence convert raw platform data into risk signals, SLA indicators, and guided next steps.

How to use this guide

Use the search bar to find any page, metric, or control. Each portal page below expands into an enterprise-level breakdown of: purpose, what you see on-screen, what each KPI means, how to operate the controls, and how to work with US Signal when action is required.

  • Search across everything: type a term like SLA, E911, deficit, MOS, or a page name to reveal matching sections.
  • Accordion design: every portal page is documented as a collapsible section. Use Expand All before exporting to PDF.
  • Contextual screenshots: each section includes thumbnails you can click to open full-size images. If your deployment hosts screenshots elsewhere, update the image URLs in the SCREENSHOTS table in the script.
  • Control Hub sourcing: metrics shown in the portal are derived from Control Hub and Webex platform APIs. The guide explains how “raw platform data” becomes operational signals.
Important note on data freshness

Most pages show a Last refresh timestamp and some run continuous polling. If values appear “Unknown” or empty, it typically indicates a permissions scope issue, an upstream API limitation, or a tenant context mismatch. US Signal can validate data paths and restore visibility using the same telemetry this portal surfaces.

Dashboard — Customer Command Center Executive posture + action routing across licenses, devices, incidents, PSTN, and adoption
Operational view

Purpose

The Dashboard is the customer’s “front door” into the portal. It condenses multiple Control Hub domains into a single operational narrative: what is healthy, what is drifting, what is broken, and what should be done next. Instead of forcing admins to pivot across Control Hub pages, export CSVs, or interpret raw platform tables, the Dashboard turns telemetry into actionable posture and direct links into the detailed pages.

What you see on this page

These elements are designed to be “executive readable” while still operationally precise.

  • Operational View banner: confirms the portal loaded Control Hub-backed data and computed the latest KPIs for the selected tenant context.
  • Action Required banner: appears when risk thresholds are exceeded (for example: license deficit detected, device offline rate elevated, or active incidents present).
  • SLA Breach Detected banner: escalates when the portal determines an incident has exceeded a defined SLA window (example: “MINOR incident has exceeded SLA threshold by X minutes”).
  • Executive Summary paragraph: auto-generated, plain-language narrative that connects signals across pages (licenses + devices + incidents + PSTN + usage) into a single story.
  • KPI tiles (30d): rapid posture snapshot (license deficit, offline devices, call volume, PSTN trunk visibility, open incidents). Each tile is a “door” to deep-dive workflows.
  • Quick Access cards: curated entry points aligned to typical operational motions: License Intelligence, Device Operations, PSTN Engineering, Call Analytics, CDR, and Incident response.
  • Tenant context chips: tenant name and role; establishes which Control Hub org is being queried and what access policy is applied.
Operational meaning

The Dashboard is where leadership can align on “what matters right now,” and where operators can immediately jump into the correct remediation workflow. In practice: if a user reports “calls failing,” the Dashboard tells you whether the most likely driver is an upstream incident, a device registration problem, a capacity/licensing constraint, or a PSTN/E911 configuration gap.

How US Signal uses this view during escalation

The portal is designed to create shared truth between customer and provider.

  • Fast triage: US Signal can validate whether the issue is tenant-local or platform-wide by correlating Dashboard signals with Status + Incidents.
  • Evidence-ready reporting: if audit questions arise, Dashboard posture provides supporting context (deficits, exposure, timeline) that can be exported from underlying pages.
  • Action routing: the “Open …” buttons move directly into the remediation domain (Open Devices, Open Licenses, View Incidents, Open PSTN).
Dashboard: executive posture + action routing across domains.
Observability — Webex Observability Suite Cross-domain risk scoring, predictive modeling, correlation, and “blast radius” posture
Correlation engine

Purpose

Observability is the portal’s risk and correlation layer. It combines signals from incidents, status, maintenance, device reachability, licensing posture, and PSTN visibility into a single stability narrative. Instead of treating each domain as independent, this page answers: “Is this tenant trending toward instability, and what will be impacted if we do nothing?”

What you see on this page

Everything here is focused on correlating domains and guiding incident readiness.

  • SLA breach detection integration: highlights when an active incident has exceeded SLA thresholds. This is critical for escalation timing and customer communication.
  • Correlation level: indicates how strongly cross-domain indicators suggest that an incident is influencing tenant experience (for example: an incident + rising offline devices + license deficits).
  • Live updates: indicates whether WebSocket-style updates are available; otherwise the portal uses hardened polling intervals.
  • Webex Observability Suite narrative: a plain-language summary that reports stability index, incident posture, license deficits, disconnected devices, predictive risk, and blast radius score.
  • Tenant Context panel: tenant selector, resolution mode, and notification controls (email target + cooldown) to reduce alert noise and prevent spam.
  • Enterprise Stability Index (0–100): aggregated stability score across incidents, status, maintenance, devices, licenses, and PSTN. Lower numbers indicate elevated risk and higher operational fragility.
  • Active Incidents tile: count + age, with SLA indicators (helps differentiate “fresh noise” vs “aging exposure”).
  • License Deficit Risk tile: correlates procurement/assignment constraints to operational risk. The portal can prioritize calling SKUs when relevant.
  • Operational Intelligence section: predictive incident modeling, blast radius scoring, AI grouping, and trends.
  • 30-Day Observability Trend: builds a rolling posture view (often using snapshot patterns or stored rollups, depending on worker capabilities).
  • Product Heatmap: shows where risk and instability concentrate by product area.
  • Incident Radar table: searchable, filterable inventory (products, severity, status, headline). Designed for rapid “what’s active and what mattered recently?” reviews.
How to interpret “blast radius”

Blast radius estimates how widely an issue can affect users and services (region, product domain, and tenant exposure). It helps decide whether to mobilize a narrow fix (single location) or a broader incident posture (multi-region communications, war room, proactive messaging).

Recommended usage patterns

Common enterprise workflows this page supports.

  • Daily posture review: check Stability Index + Active Incidents + License/Device risk tiles before business hours.
  • Incident readiness: if SLA breach risk is elevated, open Incidents to review timeline, then open Status to confirm platform scope.
  • Customer communications: use correlation strength to justify proactive stakeholder updates (especially when multiple indicators align).
Observability overview: stability index, correlation, and predictive risk posture.
Incident Radar: searchable incident inventory with product + severity context.
Licenses — License Intelligence 2.0 SKU inventory, utilization, deficits, “at risk” forecasting, history exports, and audit controls
Capacity risk

Purpose

The Licenses page is built for enterprise governance: it makes license posture obvious and makes procurement/assignment work measurable. Control Hub can show inventory, but it often does not present it in an executive-operational way. This page transforms SKU data into deficit counts, risk categories, and time-based trends so customers can prevent onboarding blocks and feature compliance issues.

Key KPIs and what they mean

These KPIs are computed from SKU totals vs consumption and mapped into risk categories.

  • License Health Score (0–100): an aggregate score where lower values indicate deficits, high utilization, or risk conditions that can impair user enablement.
  • Deficits & Risk: a concise count of SKUs that are deficient (consumed > total) and SKUs that are at risk (approaching saturation).
  • Capacity Velocity: trend readiness indicator. When enough snapshots exist, the portal estimates “days to impact” for at-risk SKUs. If insufficient history exists, it guides users to build history via periodic refresh.

What you see on this page

Designed for “exportable truth” — audit, procurement, and operational remediation workflows.

  • Tenant Context selector + Load Licenses: selects the Control Hub org context and loads license data in one controlled action.
  • Actions row: Export CSV for current view, Export 30-Day History CSV when history exists, and Clear Local History to reset client-side snapshot accumulation.
  • Auto-email controls: optional notification logic for deficits and “at risk” thresholds (with cooldown to reduce noise). Send Report Now generates an immediate summary email to the specified recipient.
  • Historical 30-Day License Health Trend: shows score trend (line), deficit count (bars), and at-risk count (dashed). This is key for leadership reporting and procurement justification.
  • Deficient SKUs table: highlights the highest severity issues with needed counts, utilization %, and severity classification.
  • License Inventory & Utilization table: full SKU inventory with filtering, sorting, utilization %, deficit value, status (OK/At Risk/Deficit), and forecast (if history exists).
  • SKU drill-down: clicking a SKU opens detail view to support “why is this happening?” analysis and remediation planning.
  • Operational Guidance section: embedded recommendations that translate findings into procurement/cleanup actions.
Enterprise value

License deficits are not just a “billing problem.” In enterprise environments they create onboarding delays, compliance gaps, and feature enablement failures. This page provides a defensible way to answer: “What will break next if we do nothing?” and “What procurement or cleanup action removes risk fastest?”

Overview: health score, deficit/risk counts, export and notification controls.
Inventory: SKU utilization, deficits, at-risk status, and drill-down.
Devices — Webex Calling Devices Fleet health, offline/stale detection, SLA breach rules, trends, and exportable device inventory
Availability

Purpose

The Devices page converts Control Hub device registration data into operational fleet health. It is designed to answer: “Are endpoints checking in, and is device reachability contributing to user-impact?” This matters because “call problems” often originate from endpoints: stale registrations, offline phones, or site-level power/network events.

Fleet summary and SLA detection

These top panels provide “instant posture” and explain the rule set being applied.

  • Fleet Summary: totals for online/offline/with issues, plus last refresh. This is the fastest way to spot a site event (spike in offline endpoints).
  • SLA / Breach Detection box: displays the rule set (example thresholds like offline ≥ 10%, stale last-seen ≥ 30 minutes, breach window) and current measurements (offline %, stale %, sustained flags).
  • 30-Day Health Trend: builds historical visibility via snapshots saved in the browser. It helps detect chronic vs transient reachability problems.

Device inventory table

This is the core operational view used during triage and remediation.

  • Filters: status filters, grouping, and search by name/model/status to isolate problem devices quickly.
  • Columns you’ll use most:
    • Status: Online/Offline and “Stale” detection (stale means last check-in exceeded the expected window).
    • Last Seen: time since the device last checked in (critical for differentiating momentary flaps vs prolonged outage).
    • Health: percent health indicator (computed) to simplify prioritization.
    • SLA: whether the device posture breaches the current SLA rule set.
    • Uptime (spark): compact sparkline trend to visualize stability without leaving the page.
    • Notes: portal-inferred guidance such as “Power/site outage likely” or “No recent check-in.”
  • Operational controls: auto-refresh, pause, export CSV, and copy JSON to share evidence with US Signal support or internal teams.
Best practice

If a high number of devices are offline, validate whether the issue is site-local (power/network) vs platform-wide (Status / Incidents). The portal supports this by linking posture across pages and providing consistent timestamps.

SLA explanation (device-level)

How the portal’s breach logic is intended to be read by enterprise operators.

Fleet: online/offline totals + SLA rule set and current measurements.
Inventory: searchable table with status, last seen, SLA posture, and notes.
Analytics — Operational Calling Analytics Adoption and reliability analytics (volume, success rate, duration, SLA status, trends, daily breakdown)
Adoption

Purpose

Analytics provides a customer-friendly operational view of calling usage and reliability outcomes. It answers: “Are we actually using the service, and how healthy are outcomes over time?” In enterprise environments, adoption metrics and failure patterns are also used to validate training effectiveness, identify underutilized sites, and spot regression after change windows.

What you see on this page

High signal, low friction: the page is built to provide clarity at a glance.

  • Top KPI tiles:
    • Total Calls (7d): counts recent activity to gauge adoption and operational baseline.
    • Success Rate: percentage of successful calls across the time window (health indicator).
    • Average Duration: helps distinguish quick failures (very short) vs stable calls (expected duration patterns).
    • SLA Status: a simplified status marker that indicates whether outcomes meet expected reliability thresholds.
  • 30-Day Call Volume Trend: sparkline-style trend to visualize growth, decline, or irregularity.
  • 7-Day Breakdown table: daily calls + failures + trend line per day, used for pinpointing which day regressions began.
  • Export CSV: produces an analyst-ready dataset for leadership reporting or deeper investigation.
Operational interpretation

If “Success Rate” is healthy but users report call quality issues, pivot to Calling Insight (quality telemetry) rather than assuming failures. If failures spike on a specific date, correlate with Maintenance windows and Incidents to distinguish platform events from configuration changes.

Analytics: volume, success rate, duration, SLA status, and trend breakdown.
CDR — Call Detail Records (Enterprise) Deep call logs, filters, troubleshooting pivots, anomalies, and export for investigation
Diagnostics

Purpose

CDR is the “investigation-grade” record view used to confirm what happened during calls: direction, endpoints, duration, results, and patterns. This page is crucial when leadership asks for proof, when engineers need to isolate failure categories, or when teams must reconcile user reports with platform truth.

What you see on this page

Designed for forensic pivots without requiring external tools.

  • Tenant Context + Load CDR: explicitly loads call records for the selected tenant and window. It surfaces record count and anomaly counts.
  • KPI tiles:
    • Total Calls: number of records loaded.
    • Failure Rate: computed from callResult categorization.
    • P95 Duration: 95th percentile duration to detect “long tail” patterns (useful for call setup delays or abnormal hold behavior).
    • SLA Status: simplified operational signal indicating whether failure/unknown rates exceed thresholds.
  • Filters & Troubleshooting Controls: time window, result category, direction, and a search box that supports caller/callee/result/ID-type pivots. Clear resets filters.
  • Call Result Breakdown: quick success vs failed vs unknown counts to validate whether the issue is systemic or isolated.
  • 30-Day Volume Trend: call volume trend to support “is this new?” analysis.
  • Top Callers / Top Called Numbers: helps identify whether the issue is concentrated to a user, a queue/AA, or a specific DID/destination.
  • CDR Records table: primary list view with pagination and row-based drilldown. Designed to support sharing: export CSV for further analysis.
When to use CDR vs Analytics

Use Analytics for posture and trends (what is happening at a high level). Use CDR for evidence and investigation (what happened on specific calls, which parties, which results). If quality is the concern (MOS, jitter, packet loss), pivot to Calling Insight where report-driven quality telemetry is interpreted.

CDR: filters, KPI tiles, breakdown charts, and record table.
Records: pagination + drilldown view for investigation workflows.
PSTN — PSTN Engineering & Architecture Topology, routing validation, misconfiguration detection, E911 compliance scoring, and exportable diagnostics
E911 + Routing

Purpose

PSTN is the portal’s architecture and compliance cockpit. It is built to make routing and E911 posture visible without requiring customers to manually chase relationships across Control Hub. The page is especially valuable for multi-location environments where DID inventory, route groups, trunks, and emergency calling configuration must be correct to avoid business and safety risk.

What you see on this page

Each panel is designed for engineering clarity and executive explainability.

  • Executive Summary: rolls up key counts (locations, trunks, DIDs), calling-enabled location posture, routing failures/warnings, and E911 compliance score.
  • Top KPI tiles:
    • Total Locations: calling-enabled footprint (often used to scope remediation).
    • Total Trunks: visibility into redundancy and trunk inventory (important for resiliency).
    • Total DIDs: inventory plus unassigned counts (helps detect configuration gaps and governance issues).
    • Capacity Score: predictive risk indicator (used to forecast whether PSTN architecture is approaching constraints).
  • Control toggles:
    • Validate Routing evaluates inferred routing paths and flags warnings/failures.
    • Detect Misconfig runs a rule engine to find common configuration mistakes.
    • Failover Simulation overlays “what if trunk fails?” assumptions onto the topology visualization.
    • Outage Simulation models service interruption impact to clarify exposure.
    • Export JSON produces an engineering-grade evidence package for US Signal or internal review.
  • E911 Compliance Scoring: reports configured vs missing E911 across calling-enabled locations, including high-risk locations.
  • Routing Path Validation: distinguishes validated vs warning vs hard failure and highlights top issues (example: E911 missing or not detectable).
  • PSTN Architecture Topology: clickable, animated paths showing how the portal infers routing relationships (customer org → PSTN type → trunk/route group/call routing → E911).
  • Trunk Utilization Heatmap: best-effort representation of trunk usage/visibility when trunk fields are present; otherwise “unknown.”
  • Predictive Capacity Forecast: local snapshot model used to project risk trajectory when server-side rollups are not available.
  • Location PSTN Architecture table: row-based drilldown showing per-location PSTN type, calling enabled, trunks, DIDs, redundancy, capacity, E911, and blast radius assumptions.
  • Misconfiguration Detection Engine: a prioritized findings list (WARN/FAIL) describing likely issues per location (example: “No trunks linked” or “E911 appears missing”).
  • Diagnostics panel: shows checks performed and endpoint paths used, allowing fast troubleshooting when fields are unavailable or permissions block visibility.
Why PSTN is “high stakes”

PSTN misconfiguration can cause silent failures (calls don’t route as expected) and can create emergency calling liability. This page is built to surface risk early, translate it into clear remediation steps, and provide exportable evidence for governance.

Calling Insight — Reports + Quality Intelligence Reports API orchestration, quality parsing (MOS/jitter/loss), heatmaps, alerts, and AI troubleshooting summary
Quality

Purpose

Calling Insight is where the portal bridges the gap between “calls succeeded” and “calls were good.” It uses the Webex Reports API flow to list existing reports, run new reports for a date range, automatically download report outputs, and parse quality fields into actionable intelligence. This is the page used when the user experience problem is quality (audio/video/media) rather than outright failure.

Report Controls (top of page)

These controls are the “pipeline” that produces quality metrics.

  • Report selector: choose a report type (example: Calling Media Quality Report).
  • Date range pickers: defines the report window for the selected tenant.
  • Run + Auto Download: creates the report and downloads the output for parsing. This is designed to minimize manual steps for operators.
  • Refresh List: reloads report inventory and updates status, especially useful when report generation is asynchronous.
  • Filter reports (title/status/created): narrows inventory to locate the exact report you want to analyze.
  • Export JSON: produces an evidence package including parsed metrics and metadata useful for escalation or long-term baselining.

Insight Views (tabs)

These tabs convert raw report rows into operator-ready visualizations.

  • Overview: primary posture view, typically with 30-day trend (MOS/jitter/loss) and heatmap by dimension.
  • Media Quality: focuses on media path quality signals (jitter, packet loss) and their distribution.
  • Calling Quality (App): isolates app-side factors and identifies likely root cause buckets.
  • Engagement: highlights how usage patterns relate to perceived experience.
  • Call Queues / Auto Attendant: provides targeted insight for enterprise call flows when report fields support it.

Inventory, parsing, and alerting

Where operators “select one report and turn it into intelligence.”

  • Reports table: shows title, status, range, created timestamp, service, and download action. Selecting a row loads details.
  • Selected Report Summary: shows the current report context and enables Load + Parse flows.
  • Parsed Report Data: a structured output area that confirms parsing success and allows downstream visualization.
  • Live Quality Alerts: alerts derived from parsed CSV metrics (MOS, jitter, packet loss, latency). If server-side rollups exist, the portal can also fetch precomputed alerts for speed and consistency.
  • AI Troubleshooting Summary: generates a concise, action-oriented interpretation of worst buckets and likely next checks. It is designed to speed triage and improve the quality of escalation tickets.
Practical example

If users say “calls connect but audio breaks up,” run a media quality report for the date range, parse MOS/jitter/loss, then use the heatmap to identify whether the problem clusters by location, user, or leg. Share the parsed output with US Signal so engineering can pinpoint whether the issue is WAN, ISP, endpoint, or platform-exposure related.

Calling Insight: reports pipeline + quality posture cards + insight tabs.
Inventory + AI: select a report, parse output, and generate an operator-ready summary.

  • Computed blast radius: expresses scope/region/status/tenant relevance.

What you see on this page

Change operations features aligned to enterprise workflows.

  • Blast Radius card: condenses scope + region + relevance for quick decision-making.

  • Predictive Window Clusters: groups similar changes by time bucket + keywords + description “fingerprint” to reduce noise and reveal patterns.

Glossary — Definitions used throughout the portal Contextual definitions to align executives and operators on consistent meaning
Reference

Key terms

These definitions are written to help both technical and non-technical stakeholders interpret the portal consistently. If your organization uses different SLA thresholds or naming conventions, US Signal can align the portal definitions to your internal governance.

SLA

A service-level expectation (time or threshold) used to classify urgency and escalation posture. In this portal, SLA is applied to incidents and to derived operational signals (device reachability, failure rates).

SLA Breaching vs Breached

“Breaching” indicates the incident is approaching or exceeding the SLA threshold; “Breached” indicates the threshold has been exceeded and escalation urgency should increase (communications + mitigations).

Stability Index

A 0–100 score that aggregates posture across incidents, status, maintenance, devices, licenses, and PSTN. Lower values indicate increased instability and higher probability of user impact.

Blast Radius

An estimate of how wide an issue can spread (region, product domain, tenant exposure). Used to decide whether to run narrow remediation or broader incident posture.

Deficit

Consumed units exceed purchased/allocated units (for example, license usage higher than total seats). Deficits can block onboarding and feature enablement.

At Risk (Licensing)

A SKU nearing saturation (example: ≥ 90–95% utilization). “At risk” is a pre-deficit warning to enable procurement or cleanup before impact occurs.

Offline vs Stale (Devices)

Offline typically indicates no registration/reachability; stale indicates the device has not checked in within the expected time window. Both can contribute to perceived calling issues.

MOS

Mean Opinion Score: a quality indicator used to estimate perceived media quality. Lower MOS indicates poorer experience; used alongside jitter/loss/latency.

Jitter

Variation in packet arrival timing. High jitter can produce choppy audio even when calls “connect successfully.”

Packet Loss

Percentage of lost packets in media flow. Loss directly degrades audio quality and can cause gaps or robotic sound.

P95

95th percentile: the value below which 95% of observations fall. Used to expose “long tail” behaviors (for example, unusually long durations or delays).

E911

Emergency calling configuration required to ensure emergency services receive accurate callback and location information. Missing E911 is a high-risk compliance and safety issue.

DID

Direct Inward Dial number inventory. Used for routing, assignment, and governance. Unassigned DIDs often indicate configuration gaps.

Reports API

Webex reporting mechanism used to generate and retrieve report outputs (often CSV). Calling Insight uses this workflow to compute quality posture from report fields.
