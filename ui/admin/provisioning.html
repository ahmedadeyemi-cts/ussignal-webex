Overview & SLA

This page documents US Signal’s standard operating procedure to provision a new Webex customer (Webex Calling and Webex Meetings). The runbook is written for repeatability, accountability, and audit evidence. It assumes the provisioning request is received at webexadminteam@ussignal.com and that provisioning is completed within 24 hours of email receipt (unless blocked by missing prerequisites).

SLA Commitment
≤ 24 hours
From intake email received to baseline tenant configured and validated.
Common Blockers
Identity / PSTN
Domain ownership, SSO readiness, number porting, or PSTN selection can delay completion.
Audit Artifacts
Evidence Pack
Screenshots, exported configs, ticket IDs, and change approvals linked in Evidence Tracker.

Audit expectation

Each provisioning action should be traceable to a request (email/ticket), an approver (if required), an operator, and evidence (Control Hub screenshots/exports). If an item is intentionally deferred (for example SSO or PSTN porting), document the deferral and owner in the Evidence Tracker.

1) Intake: Provisioning Email

Provisioning begins when US Signal receives the initial request email at webexadminteam@ussignal.com. This email is treated as the system of record for the initial request and is associated to an internal ticket (or equivalent work item) to ensure traceability.

T+0 (Email received)
SLA starts
Evidence: email message ID + ticket ID

Create the provisioning ticket

  • Open an internal ticket titled: “Webex Provisioning — <Customer Name>”.
  • Attach the original email (or reference message ID), include purchase details and requested services (Calling + Meetings).
  • Assign an owner and an accountable reviewer (two-person integrity for audit readiness).
T+0–30m
Validate inputs
Evidence: intake checklist

Capture required provisioning inputs

  • Customer legal name and primary admin contact (name, email, phone).
  • Requested license quantities (Calling, Meetings, add-ons as sold).
  • Customer timezone, main site address, and any location requirements.
  • PSTN intent (Cloud Connected PSTN, Local Gateway, or existing provider as sold).
Intake checklist (required)⌄
Minimum evidence to capture⌄
  • Email receipt timestamp (source of truth).
  • Ticket ID and assignment history.
  • Order/quote reference (where applicable).
  • Initial Control Hub org view screenshot once available.

2) Triage & Validation

Triage ensures we can meet the 24-hour provisioning goal. The objective is to identify blockers early (identity, domain, PSTN/numbering, compliance requirements) and either resolve them immediately or document them as explicit dependencies with an owner and target date.

Triage decision points⌄
Decision pointWhat we verifyIf not ready
Identity readinessCustomer domain(s), preferred identity provider, directory sync expectations.Proceed with local admin(s) and document SSO as a post-provision task.
PSTN readinessCustomer’s PSTN model (CCP, Local Gateway, porting timeline).Provision tenant and Calling baseline; defer numbers/porting with documented plan.
Compliance constraintsRetention, legal hold, eDiscovery, recording, data residency expectations.Apply baseline security, then schedule compliance tuning with customer stakeholders.

Triage checklist (recommended within first 2 hours)⌄

3) Create / Verify Tenant

The provisioning team ensures the customer organization (tenant) exists in Webex Control Hub, that US Signal has the appropriate administrative access, and that the baseline organization settings are ready for licensing and service configuration.

Change control

If creating or modifying org-level settings that materially impact security or user experience, record the ticket ID and capture screenshots or exports showing the before/after state.

Baseline org setup actions⌄
  • Confirm the customer org is present in Control Hub and that the primary customer admin is associated.
  • Validate US Signal administrative access is correct for the support model (partner/admin relationship as applicable).
  • Confirm organization naming, timezone, and default locale (if used for calling locations).
  • Establish a standard documentation location for evidence artifacts (ticket attachments, secure file share, etc.).
Create/Verify checklist⌄

4) Control Hub: Initial License Setup

Licensing is configured early because it enables downstream setup (user creation, Calling services, Meetings scheduling capabilities, and device onboarding). The provisioning team assigns only the licenses purchased and documents the entitlement state.

License configuration objectives⌄
  • Confirm purchased entitlements for Webex Calling and Webex Meetings.
  • Establish a standard license assignment policy (manual, template-based, or directory-driven).
  • Assign initial admin/service accounts required for setup and validation.
  • Capture evidence: license counts, assigned users, and subscription view.
License checklist⌄

5) Webex Calling Baseline Configuration

Calling setup is performed according to what was sold and the customer’s design decisions. The baseline configuration ensures the tenant is ready for numbers, locations, routing, and device/user activation without creating security or operational risk.

Baseline items (typical)⌄
  • Create initial locations with correct addresses and timezones (required for emergency calling and services).
  • Confirm calling settings: dialing preferences, caller ID defaults, and user calling permissions baseline.
  • Prepare number strategy: initial numbers (if available), porting plan, or placeholder routing if porting is pending.
  • Define baseline call routing objects as applicable: auto attendants, hunt groups, call queues (as sold / requested).
  • Record evidence: location list, emergency settings status, and initial routing objects.
Calling checklist⌄

6) Webex Meetings Baseline Configuration

Meetings setup focuses on enabling the customer’s baseline meeting experience while applying sensible security defaults. If the customer has strict policies (external collaboration restrictions, recording policy, or retention rules), capture approvals and evidence.

Baseline items (typical)⌄
  • Confirm meeting site settings and default security: lobby/host controls, guest access posture, and sharing defaults.
  • Confirm recording posture (enabled/disabled), storage expectations, and governance requirements.
  • Validate at least one licensed user can schedule and start a meeting end-to-end.
  • Document any requested exceptions (for example allowing external participants) with ticket approvals.
Meetings checklist⌄

7) Security & Compliance Baseline

Security controls are applied with an “audit-first” posture: minimize exposure, enforce strong identity controls, and ensure actions are traceable. If the customer requires deviations, the deviation must be approved and captured as evidence.

Baseline control themes⌄
  • Identity hardening: enforce MFA where applicable, limit admin roles to least privilege.
  • External collaboration posture: define how guests/external users can interact.
  • Data governance: retention expectations, recording policies, and compliance tooling as required.
  • Operational logging: ensure changes are tracked via ticketing and evidence capture.
Security checklist⌄

8) Identity: Domains, SSO, and Directory Synchronization (as applicable)

Identity integration is often the biggest time variable. The tenant can still be provisioned within 24 hours using local admins, while SSO and directory sync are implemented as a planned follow-on activity with explicit milestones.

Identity options and audit guidance⌄
  • Domain verification: confirm customer ownership of domains before enabling domain-based policies.
  • SSO: implement only after customer provides IdP metadata and validates test accounts.
  • Directory sync: align attribute mapping and group-based license assignment strategy.
  • Evidence: domain verification records, SSO change approvals, and test results.
Identity checklist (optional during initial 24 hours)⌄

9) Devices and Endpoints (as applicable)

Device readiness is configured based on what the customer purchased (desk phones, room devices, soft clients). The baseline goal is to ensure devices can be onboarded securely and managed consistently.

Device readiness items⌄
  • Define device onboarding method: activation codes, MAC-based, or other approved approach.
  • Establish baseline device policies and admin roles for device management.
  • Verify at least one test device path (phone or app) can register and place calls (if Calling is enabled).
  • Capture evidence: device policy screenshots and successful registration proof.
Devices checklist⌄

10) PSTN / Connectivity

PSTN configuration depends on the customer’s purchased PSTN model. During initial provisioning, the objective is to ensure Control Hub is configured to support the intended path and that dependencies such as number porting are tracked.

PSTN provisioning paths⌄
ModelWhat we configureTypical dependencies
Cloud Connected PSTNEnable CCP for locations, align numbering plan and emergency calling requirements.Carrier onboarding, address validation, number assignment/ordering.
Local GatewayPrepare trunking strategy and location config; align to customer SBC/CUBE readiness.CUBE deployment, certificates, routing design, firewall, testing windows.
Existing providerDocument the interconnect approach and controls; configure as supported by design.Provider details, number porting timelines, test numbers.
PSTN checklist⌄

11) Validation & Acceptance Testing

Validation confirms the tenant is functional and safe to hand off. At minimum, validate one Calling flow (if Calling is enabled) and one Meetings flow, and document results as evidence.

Minimum acceptance tests (baseline)⌄
  • Control Hub access: customer admin can sign in and view required sections.
  • Licensing: licensed user is active and services show as provisioned.
  • Meetings: schedule/start/join meeting; verify audio/video/sharing.
  • Calling: register endpoint; place a test call (internal or PSTN as available).
  • Security posture: verify baseline policies are applied and documented.
Validation checklist⌄

12) Customer Handoff (Provisioning Complete)

Provisioning is considered complete when the baseline tenant configuration is applied, validation tests are documented, and the customer receives a handoff message detailing what is ready now and what is scheduled next.

Handoff contents (what the customer receives)⌄
  • Confirmation that Webex Calling and Webex Meetings are provisioned and ready for user onboarding.
  • Summary of what was configured (licenses, baseline policies, locations, PSTN readiness).
  • List of open items (SSO, directory sync, number porting, advanced routing objects) with owners and dates.
  • Support contacts and escalation instructions.
Handoff checklist⌄
